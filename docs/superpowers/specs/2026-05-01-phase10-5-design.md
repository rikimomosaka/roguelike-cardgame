# Phase 10.5 — カードテキスト自動文章化 + 開発者編集ツール 設計

> 作成日: 2026-05-01
> 対象フェーズ: Phase 10.5（Phase 10.3-MVP の視覚 UX 完了後に挿入）
> 親ロードマップ: [`2026-04-20-roadmap.md`](../plans/2026-04-20-roadmap.md) — Phase 10.5 セクション

## ゴール

カード（および将来的にレリック / ポーション / 敵 / ユニット）の `description` / `upgradedDescription` を **`effects` 配列から自動文章化**し、JSON への手書きを廃止する。同時に、ブラウザ上で **JSON のスペック調整・新規作成・バージョン切替**ができる **開発者専用ツール** を整備し、テストプレイでの即時反映と、本番ビルドへの誤伝播防止を両立する。

**完成イメージ**:
- カード追加時、`effects` を書けば `description` が自動生成される（`description` を JSON に書けばオーバーライド）
- 開発中に `?dev=1` 等で開発者メニューが開き、カードのスペックを編集 → ホットリロードで即反映
- 編集はソース JSON に **`versions[]`** として履歴蓄積、active version だけ runtime で参照
- 「実験 version」は `data-local/dev-overrides/` でローカル完結、満足したら "Promote" でソース JSON に転記
- 本番ビルドでは開発メニューが UI ・ Server endpoint の双方で物理的に存在しない

## 完了判定（Phase 10.5 全体）

- `dotnet build` 警告 0 / エラー 0
- `dotnet test` 全テスト緑（既存 + Phase 10.5 新規 formatter + versioned loader テスト）
- Client `npm run build` がビルドエラーなし
- カードカタログ全エントリの `description` がローダー時点で auto-text と一致するか、または JSON 内 override で意図的に異なる
- 開発者メニュー (`?dev=1`) でカード一覧表示、編集、新規作成、version 切替、Promote が動作
- 本番ビルド (`vite build` 後の dist) に dev メニューコードが含まれていないことを bundle 検査で確認
- 本番 ASP.NET Core 環境 (`Production`) で dev endpoint が 404 を返すことを確認
- 移行スクリプト実行後、全 card JSON が versioned 形式に変換されており、ローダーが新形式を読める
- relic / potion / enemy / unit にも同じ仕組みが拡張完了

各 sub-phase の完了判定は対応する plan 文書で詳細化する。

### 1-3. 既存 description 手書きとの関係

| ケース | 挙動 |
|---|---|
| JSON `description` が空 / null | formatter で自動生成、それを runtime で使用 |
| JSON `description` が非空 | JSON の手書き値を優先 (override) |
| 開発ツールで「auto に切替」ボタン | JSON の `description` を null に設定 |
| 開発ツールで「手動 override」 | 編集した文字列を `description` に保存 |

formatter は spec 純関数なので、override が無ければ毎回同じ文字列を吐く。テストでスナップショット可能。

## 2. CardTextFormatter (10.5.A の中核)

### 2-1. API

```csharp
namespace RoguelikeCardGame.Core.Cards;

public static class CardTextFormatter
{
    /// <summary>
    /// CardDefinition と upgraded フラグから表示テキストを生成する。
    /// JSON description が非空なら formatter は呼ばれず override が使われる前提。
    /// </summary>
    public static string Format(CardDefinition def, bool upgraded);

    /// <summary>
    /// effects 配列単体からの生成 (テスト・dev tool プレビュー用)。
    /// </summary>
    public static string FormatEffects(IReadOnlyList<CardEffect> effects);
}
```

### 2-2. effect プリミティブ別のテンプレート

各 effect → 1 文（句点で終わる）に変換し、改行で連結する。

| action | scope | テンプレート (JP) |
|---|---|---|
| attack | single | `敵 1 体に {amount} ダメージ。` |
| attack | random | `敵ランダム 1 体に {amount} ダメージ。` |
| attack | all | `敵全体に {amount} ダメージ。` |
| block | self | `ブロック {amount} を得る。` |
| block | ally(single) | `味方 1 体にブロック {amount} を与える。` |
| block | ally(all) | `味方全体にブロック {amount} を付与。` |
| draw | self | `カードを {amount} 枚引く。` |
| discard | self | `手札 {amount} 枚を捨てる。` |
| debuff | single,enemy | `敵 1 体に {name} {amount}。` |
| debuff | all,enemy | `敵全体に {name} {amount}。` |
| buff | self | `自身に {name} {amount}。` |
| buff | all,ally | `味方全体に {name} {amount}。` |
| heal | self | `HP を {amount} 回復。` |
| heal | single,ally | `味方 1 体の HP を {amount} 回復。` |
| summon | self | `{name} を召喚 (lifetime {lifetime})。` |

ステータス名 (weak / vulnerable / strength / dexterity / poison / omnistrike) は **対応する日本語名へ自動翻訳**（フォーマッタ内部のマッピング）：

| status id | JP |
|---|---|
| weak | 脱力 |
| vulnerable | 脆弱 |
| strength | 筋力 |
| dexterity | 敏捷 |
| poison | 毒 |
| omnistrike | 全体攻撃 |

未知のステータス id はそのまま英語表示（後日テンプレート追加で対応）。

### 2-3. 連結ルール

- 複数 effects は **句点 + 改行** で連結 (`「.」 + "\n"`)
- 同一 action の連続（例: attack 3 連） → `× 3 回` 形式に集約: `敵 1 体に {amount} ダメージ × {hits} 回。`
- conditional / chained 効果は当面個別文として並べる（将来「条件: X → Y」構文を追加する余地）

### 2-4. テスト戦略

- **単体テスト**: 各 effect プリミティブ × scope 組み合わせを網羅
- **スナップショットテスト**: 既存 ~30 カード全部について `formatter(def, false) === def.Description` と `formatter(def, true) === def.UpgradedDescription` を検証
  - スナップショット不一致時はテスト fail し、開発者が「formatter テンプレ修正」or「JSON description 削除して auto に乗る」を判断
- **i18n 余地**: locale 引数を取らない当面 JP 固定だが、内部関数を `static class JpTemplates` 等で分離して将来差替え可能に保つ

### 2-5. 連携: Server DTO 変換

現状 `BattleStateDtoMapper` / `CatalogController` などが `def.Description` をそのまま渡している。10.5.A 完了時には:

```csharp
string descFor(CardDefinition def, bool upgraded)
{
    var manual = upgraded ? def.UpgradedDescription : def.Description;
    if (!string.IsNullOrWhiteSpace(manual)) return manual;
    return CardTextFormatter.Format(def, upgraded);
}
```

を一箇所のヘルパに集約し、DTO・図鑑・報酬全部から呼ぶ。

## 3. Versioned JSON Schema (10.5.B)

### 3-1. ファイル構造

旧 (flat):
```json
{
  "id": "strike",
  "name": "ストライク",
  "rarity": 0,
  "cardType": "Attack",
  "cost": 1,
  "effects": [...],
  "upgradedEffects": [...],
  "description": null,
  "upgradedDescription": null
}
```

新 (versioned):
```json
{
  "id": "strike",
  "name": "ストライク",
  "displayName": null,
  "activeVersion": "v1",
  "versions": [
    {
      "version": "v1",
      "createdAt": "2026-05-01T00:00:00Z",
      "label": "original",
      "spec": {
        "rarity": 0,
        "cardType": "Attack",
        "cost": 1,
        "upgradedCost": null,
        "upgradable": true,
        "effects": [{ "action": "attack", "scope": "single", "side": "enemy", "amount": 6 }],
        "upgradedEffects": [{ "action": "attack", "scope": "single", "side": "enemy", "amount": 9 }],
        "description": null,
        "upgradedDescription": null
      }
    }
  ]
}
```

### 3-2. ローダーの責務 (`CardJsonLoader` v2)

```csharp
public static CardDefinition LoadFromJson(string json)
{
    var versioned = ParseVersioned(json);
    var active = versioned.Versions.First(v => v.Version == versioned.ActiveVersion);
    return new CardDefinition(
        Id: versioned.Id,
        Name: versioned.Name,
        DisplayName: versioned.DisplayName,
        Rarity: active.Spec.Rarity,
        CardType: active.Spec.CardType,
        Cost: active.Spec.Cost,
        // ... 残りは active.Spec から
    );
}
```

旧形式互換:
- `versions` キーが無ければ root を spec として `[{ version: "v1", spec: root }]` 相当に内包扱い
- 新規 JSON は必ず versioned 形式で生成

### 3-3. 自動採番ルール

- 新 version 作成時: 既存 versions の中から `v\d+` パターン抽出 → 最大 N + 1
- 例: `["v1", "v2", "v5"]` → 次は `v6`（gap は埋めない）

### 3-4. 移行スクリプト

`tools/migrate-to-versioned-cards.ps1` (or .csx) を一回実行:
1. `src/Core/Data/Cards/*.json` を読む
2. 各ファイル root を `spec` として `versions[0]` に格納、`activeVersion: "v1"` をセット
3. `createdAt` は今日の日付、`label: "original"` をハードコード
4. 元の id / name / displayName は外側に残す
5. ファイル上書き

実行後、`git diff` で全カードが versioned 化されたことを確認 → 一発 commit。

## 4. Override 層 (10.5.B 後段 + 10.5.D)

### 4-1. ディレクトリ

```
src/Core/Data/Cards/strike.json          ← committed
data-local/dev-overrides/cards/strike.json ← gitignored
```

`data-local/dev-overrides/` はあらかじめ `.gitignore` に追加。

### 4-2. override JSON の構造

base と同じ versioned 形式。**完全な置換ではなく、versions[] に追加 / activeVersion ポインタ上書き** の差分構造。

```json
{
  "id": "strike",
  "activeVersion": "v3",
  "versions": [
    {
      "version": "v3",
      "createdAt": "2026-05-02T10:00:00Z",
      "label": "EXPERIMENT-cost0",
      "spec": { "...": "...", "cost": 0 }
    }
  ]
}
```

`name` / `displayName` は base 側に必ずあるので override では省略可。

### 4-3. マージ規則

DEV モード (`IsDevelopment()`) のみ:
1. base から versioned オブジェクトをロード
2. override が同 id で存在すれば:
   - `versions[]` を union（base ∪ override、id 衝突時は override 優先）
   - `activeVersion` を override の値で上書き
3. マージ後の active version の spec で `CardDefinition` を組み立てる

非 DEV モード:
- override は読まない（dir 自体を ignore）

### 4-4. Promote (昇格)

開発ツールで「Promote v3 to source」を押す:
1. override の v3 entry を base JSON の `versions[]` に追加
2. オプションで base.activeVersion も v3 に切替
3. override から v3 entry を削除（移動扱い）
4. `data-local/backups/cards/strike-{timestamp}.json` に base 直前内容をバックアップ

→ 開発者は `git diff src/Core/Data/Cards/strike.json` で変更確認 → commit。

## 5. 開発者メニュー (10.5.C–E)

### 5-1. 三段ゲート

### 5-4. JSON エディタ (10.5.D)

- MVP: `<textarea>` + `JSON.parse` + Ajv で schema 検証
- エラー時は赤線 + メッセージ表示、保存ボタン disabled
- schema は `src/Core/Cards/schema/card-spec.json` に定義（formatter / loader と共有）
- 構造化フォーム（dropdown 等）は **将来拡張**、本フェーズではスコープ外

### 5-5. Server endpoint (DEV gate 内)

```
GET    /api/dev/cards               カード一覧 (versioned 形式)
GET    /api/dev/cards/{id}          個別カード詳細
POST   /api/dev/cards/{id}/versions {label, spec} → 新 version 追加 (override に書込)
PATCH  /api/dev/cards/{id}/active   {version} → activeVersion 切替 (override に書込)
DELETE /api/dev/cards/{id}/versions/{ver} version 削除 (override 内のみ削除可)
POST   /api/dev/cards/{id}/promote  {version} → override から base に転記
POST   /api/dev/cards/new           新規カード作成 (override に新規 base 相当として作成 → promote で source へ)
```

すべて `app.MapWhen(env.IsDevelopment, ...)` 配下。

### 5-6. Hot reload

- Server: `FileSystemWatcher` で `data-local/dev-overrides/cards` を監視 → 変更検知で `DataCatalog` を再構築
- Client: 編集保存後に catalog API を再 fetch（既存の `useCardCatalog` hook が依存するエンドポイント）
- 戦闘中に catalog が変わるリスクは「戦闘開始時にスナップショット」で吸収可能（既存 `BattleEngine.Start` が catalog を引数で受けている設計）

## 6. relic / potion / enemy / unit への拡張 (10.5.F)

### 6-1. 共通テンプレート

card と同じパターンを 4 種別に展開:
- `RelicTextFormatter` / `PotionTextFormatter` / `EnemyMoveTextFormatter` / `UnitTextFormatter`
- versioned JSON schema は同じ構造（`activeVersion` + `versions[]` + `spec`）
- override dir は `data-local/dev-overrides/{relics|potions|enemies|units}/`
- dev menu に種別タブを追加

### 6-2. 種別固有の差分

| 種別 | 主な差分 |
|---|---|
| relic | `triggers[]` の自動文章化 (OnTurnStart / OnEnemyDeath etc.) |
| potion | `effects[]` は card と同じプリミティブだが「使い切り」で文末の言い回し変更 |
| enemy | `moves[]` 配列ごとに展開、ターン予告の「〜する」表記 |
| unit | move + lifetime 表記。enemy と共通テンプレ多め |

formatter は種別ごとに別クラスだが、共通の effect プリミティブ部分は内部で `EffectTextHelpers` 等に共通化。

## 7. 非ゴール (本フェーズスコープ外)

- 本格的な i18n (英語版テキスト)
- 構造化フォームエディタ (dropdown / number spinner)
- DB 永続化 (override は file-system のみ)

- 自動文章化のローカライズ済み翻訳機械学習活用（テンプレートだけ）

## 8. 既存テスト・データへの影響

### 8-1. 影響範囲

- **Core**: `CardJsonLoader` を versioned 対応に書き換え。既存テスト (`Core.Tests/Cards/CardJsonLoaderTests.cs` 等) は flat 形式の JSON 文字列をテストフィクスチャに持つので、versioned 形式に書き換える必要あり
- **データ**: `src/Core/Data/Cards/*.json` 全ファイル (現状 ~40 ファイル) を移行スクリプトで一括変換
- **Server**: catalog endpoint の DTO は変わらない (active version の spec を返すだけ)
- **Client**: 表示側は変更なし (description が auto-text に変わる以外)

### 8-2. 既存 description との比較

10.5.A 完了時に **全 ~40 カードについて auto-text と既存手書きが一致するか** をスナップショットで検証。差異がある場合、各カードに対して以下から選択:
1. formatter テンプレを微修正して既存手書きに合わせる
2. JSON `description` 手書き値を `null` にして auto に乗せる
3. 個別カードの特殊文言として手書き値を残す（override）

差異の数だけ判断ポイント。本フェーズはこれを設計上の作業項目として明示する（plan 文書で件数管理）。

## 9. 段階的実装方針

### 9-1. 着手順序

1. **10.5.A** から始める（formatter のみ、既存仕組みに乗る）
2. 各 sub-phase が独立 mergeable
3. 10.5.A の TDD plan を別文書 `2026-05-01-phase10-5A-formatter.md` に切る
4. 後続 sub-phase は前段 merge 後に新規 plan 文書を起こす

### 9-2. 10.5.A の最小スコープ確認

10.5.A は **versioning も dev tool も触らない**。CardTextFormatter を作って、Server DTO で auto-text を反映するだけ。これにより:
- 10.5.B 以降の前提（formatter が動く）が早く確立
- ロールバック容易（formatter 削除 + DTO 変換差分 revert）
- 視覚的成果はゲーム内 description が一部変化することのみ。挙動は不変

### 9-3. 10.5.A 後のチェックポイント

- 全カードの description 表示が auto-text に変わっても違和感ないか手動確認
- スナップショットテストが緑
- 続いて 10.5.B (versioning + 移行) に進むか、別優先タスク (立ち絵投入 等) を挟むかをここで判断

## 10. 用語

- **base JSON**: `src/Core/Data/{Cards,Relics,Potions,Enemies,Units}/{id}.json` (commit 対象)
- **override JSON**: `data-local/dev-overrides/{...}/{id}.json` (gitignored)
- **active version**: `activeVersion` フィールドが指す versions[] 内のエントリ。runtime で実際に使われる
- **promote**: override の version entry を base JSON に転記する操作
- **auto-text**: `CardTextFormatter` 等が自動生成した description 文字列
- **manual override**: JSON `spec.description` に書かれた手書き文字列（auto より優先）
