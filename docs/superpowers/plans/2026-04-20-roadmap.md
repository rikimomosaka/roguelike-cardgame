# Roguelike Card Game 実装ロードマップ

> **このドキュメントはロードマップ（上位計画）です。** 各フェーズの実装は本ファイルとは別に `docs/superpowers/plans/2026-04-20-phaseNN-<name>.md` として TDD 粒度の詳細計画を書き起こしてから着手します。詳細計画が存在するフェーズのみ着手可能。

**Goal:** Slay the Spire 風のソロ／協力両対応ローグライク（PvE）を、まずは「バトル以外」で一周できる状態まで仕上げ、その後に本格バトルを差し込む。

**Architecture:**
- `Core`（C# .NET 10, 純粋ロジック, Udon# 移植対象）に全ゲームルール／状態遷移／乱数／セーブデータ構造を集約。
- `Server`（ASP.NET Core 10 + SignalR）は Core を呼ぶ薄いアダプタ。シングルプレイでも同一エンドポイント経由。
- `Client`（React 19 + TypeScript + Vite）は表示とユーザー入力のみ。状態は Server から取得。
- マスター定義（カード／レリック／ポーション／モンスター）は `src/Core/Data/*.json` にまとめ、Core 起動時に読み込む。

**Tech Stack:** C# .NET 10, ASP.NET Core 10, SignalR, xUnit, React 19, TypeScript, Vite

**前提ルール（全フェーズ共通）:**
- Core は ASP.NET Core / ネットワーク / ファイル I/O への依存を持たない。JSON 読み込みは「文字列 → モデル」のパース関数として実装し、I/O は呼び出し側で行う。
- record ベースの不変データ。状態遷移は「現在の状態 ＋ 入力 → 新しい状態」の純関数として実装する。
- 乱数は `IRng`（インターフェース）を注入する。テストでは決定的 RNG を使用する。
- テストファースト。まず失敗テスト → 実装 → テスト通過 → コミット。

## 仕様サマリ（ご提示内容を Core モデルに落とし込む前提で整理）

### ゲームループ
1. 起動 → ログイン画面
2. メイン画面：`シングルプレイ` / `マルチプレイ` / `設定` / `実績` / `終了`
3. シングルプレイ選択 → 前回セーブがあれば再開、無ければ新規ラン開始
4. ダンジョン = 3 層。各層はスタート → 分岐路（各ルート 15 マス） → ボス。
5. 各マスで種別に応じたイベント。カード／レリック／ポーション／ゴールドでデッキを強化。
6. 3 層のボスを倒すとクリア。HP 0 でゲームオーバー。どちらも最終結果を保存。

### マス種別
- `?` ランダム（選択式イベント or 商人／宝箱／休憩／敵マスと同等）
- `商人` カード／レリック／ポーションをランダム販売、有償で 1 枚カード廃棄（各マス 1 回）
- `宝箱` ゴールド＋レリック
- `休憩` HP 回復 or カード 1 枚強化
- `敵` 層の通常敵（同層最初の 3 戦は弱プール、4 戦目以降は強プール）
- `エリート敵` 層のエリートプール
- `ボス` 各層終点固定

### 戦闘（暫定・自動勝利）
- 戦闘開始時に現在デッキを「ディープコピー」してバトル用デッキを作り、バトル用デッキに対して戦闘中の加除を行う。
- バトル終了時にバトル用デッキは破棄し、ラン側デッキには影響を与えない。
- バトル以外（休憩の強化／商人の廃棄／イベントでの消去）はラン側デッキを直接変更する。
- **本フェーズではバトル処理を行わず即勝利扱い**とし、通常戦闘／エリート／ボス全てで同じ報酬フローを実行する。

### 報酬
- 敵撃破：ゴールド、ポーション（出現率に従い 0〜1 個）、カード報酬（3 択から 1 枚まで、スキップ可）。
- 宝箱：ゴールド＋レリック 1 個。

### ポーション
- 使い切り。戦闘中・非戦闘中を問わず、発動条件を満たせば使用可。使用で消費。

### レリック
- 入手時効果のみのものと、所持中常時効果のものが混在。消去・変化しない限り永続。

### セーブ
- 自動セーブ。トリガは「マス選択 → そのマスのイベント確定時点」。
- クリア／ゲームオーバー時も保存。
- ダンジョン開始後のプレイ時間を秒単位で保存。再開時は続きから加算。
- ログイン単位で 1 つのセーブを保持（現状ローカル、将来クラウド化）。

### 初期デッキ
- 固定 10 枚からスタート（内容は後で定義）。

### メニュー
- メイン画面 → 設定：マスター音量 / BGM / SE / 環境音。
- メイン画面 → 実績：カード／レリック／ポーション／モンスター図鑑、プレイ履歴。
- ダンジョン内メニュー：あきらめる（ラン放棄）、メニューに戻る（最終保存状態から再開可）、音量設定。

### マルチプレイ
- ホスト：合言葉形式でロビー作成。
- ゲスト：合言葉入力で既存ロビー参加。
- 具体的な同期ルールは本格バトル実装時に合わせて詰める（Phase 10）。

## Phase 0 — JSON データ定義と Core データモデル

**目的:** カード／レリック／ポーション／モンスター／マスタ JSON スキーマを決め、Core にロードする仕組みを作る。

**成果物:**
- `src/Core/Data/Cards/` 配下にカード JSON（初期デッキ 10 枚分＋基本プール最小限）
- `src/Core/Data/Relics/` 配下にレリック JSON（入手時効果型と永続効果型を各 2 件以上）
- `src/Core/Data/Potions/` 配下にポーション JSON（数点）
- `src/Core/Data/Enemies/` 配下に敵 JSON（1 層の弱／強プール＋エリート＋ボス）
- `src/Core/Cards/CardDefinition.cs`（record）
- `src/Core/Relics/RelicDefinition.cs`（record）
- `src/Core/Potions/PotionDefinition.cs`（record）
- `src/Core/Enemy/EnemyDefinition.cs`（record）
- `src/Core/Data/DataCatalog.cs`（文字列群を受け取り各 Definition 辞書を生成する純粋関数）
- `tests/Core.Tests/Data/DataCatalogTests.cs`

**ゲームデザイン決定事項（このフェーズで固める）:**

**カード定義（ユーザー決定）:**
- `id`: 一意の英数字 ID（内部の呼び出し・検索はすべてこれを使う）。
- `name`: カード名。
- `displayName`: 表示名。任意。通常カードでは null、プロモや同一カードのスキン違いで差し替える。未設定なら `name` を表示。
- `rarity`: `0=プロモ / 1=コモン / 2=レア / 3=エピック / 4=レジェンダリー`。
- `cardType`: `Unit / Attack / Skill / Power`。
- `cost`: 通常 0〜3、4 以上も許容。`null` 可（`null` の場合はプレイ不可＝条件付きでしかプレイされないカードを表現）。
- `effects`: 効果プリミティブの配列（例: `[{"type":"damage","amount":6}]`）。本フェーズでは実行はしないが、JSON スキーマと C# モデルだけ先に決める。
- `upgradedEffects`: 強化時の効果配列。未設定なら強化不可。

**初期デッキ（ユーザー決定）— 計 10 枚:**
- `Strike` ×5：`cardType=Attack` / `cost=1` / `effects=[damage:6]` / `upgradedEffects=[damage:9]`
- `Defend` ×5：`cardType=Skill` / `cost=1` / `effects=[gainBlock:5]` / `upgradedEffects=[gainBlock:8]`

**レリック:**
- `trigger`（`onPickup` / `passive` / `onBattleStart` / `onMapTileResolved` 等）と `effect` の組で表現。

**モンスター:**
- HP レンジ、行動テーブル（後で使う）、出現プール（`act1_weak` / `act1_strong` / `act1_elite` / `act1_boss` 等）をタグで保持。

**Done 判定:**
- `DataCatalog.LoadFromStrings(...)` に JSON 文字列を渡して全種の辞書が得られるテストが通る。
- 未知のフィールドを無視し、壊れた JSON では明示的な例外／`Result` を返すテストが通る。

## Phase 3 — ダンジョンマップ生成

**目的:** 「スタート → 分岐路 15 マス → ボス」を決定的に生成する Core ライブラリを実装。詳細は [`docs/superpowers/specs/2026-04-21-phase03-map-generator-design.md`](../specs/2026-04-21-phase03-map-generator-design.md)。

**成果物:**
- `src/Core/Random/IRng.cs` / `SystemRng.cs` / `FakeRng.cs`（乱数抽象化、Core 共通で以後も再利用）
- `src/Core/Map/TileKind.cs`（`Start` / `Enemy` / `Elite` / `Rest` / `Merchant` / `Treasure` / `Unknown` / `Boss`）
- `src/Core/Map/MapNode.cs`（`Id` / `Row` / `Column` / `Kind` / `OutgoingNodeIds`）
- `src/Core/Map/DungeonMap.cs`（`Nodes` / `StartNodeId` / `BossNodeId`）
- `src/Core/Map/MapGenerationConfig.cs` 群（`EdgeCountWeights` / `TileDistributionRule` / `FixedRowRule` / `RowKindExclusion` / `PathConstraintRule` / `IntRange` / `TileKindPair`）
- `src/Core/Map/MapGenerationConfigLoader.cs`（埋め込み JSON → `MapGenerationConfig`）
- `src/Core/Map/Config/map-act1.json`（Act 1 用既定値、埋め込みリソース）
- `src/Core/Map/DungeonMapGenerator.cs`（`IDungeonMapGenerator` 実装） / `MapGenerationException.cs`
- `src/Server/Program.cs` に `IDungeonMapGenerator` と `MapGenerationConfig` の DI 登録
- `tests/Core.Tests/Map/` 配下のテスト一式（決定性、エッジ規則、FixedRow/Exclusion、エッジ数分布、ルート制約、再生成上限、Config Loader）

**トポロジー（このフェーズで確定）:**
- 固定 5 列グリッド（列 0..4）、Row 0=Start（列 2）、Row 1..15=中間、Row 16=Boss（列 2）。
- 中間各行のノード数は `[RowNodeCountMin, RowNodeCountMax]` = `[2, 4]` の一様乱数。
- エッジは Row r → Row r+1 の一方向、かつ `|col diff| <= 1` の ±1 隣接のみ（例外：Start→Row 1 と Row 15→Boss は列制約なし）。
- 1 ノードの出次数は 1..3、`EdgeCountWeights = w1:w2:w3 = 82:16:2` で 1 エッジを優先。
- 同じ seed + 同じ config から常に同一のマップが生成される（決定性）。

**タイル配置制約（このフェーズで確定）:**
- **Row 9 = 宝箱（全ノード固定）**：行のノード数は他行と同じく乱数、ただし種別は全て Treasure。ルートが 9 マス目で合流せず、分岐を保持したまま進める。
- **Row 15 = 休憩（全ノード固定）**：同上、種別は Rest。
- **Row 1 = 敵固定**：Start 直後は必ず敵戦闘。
- **Row 14 ≠ 休憩**：Row 15 と連続 Rest になるため。
- **Elite は Row 6 以降**：序盤強敵回避（`MinEliteRow = 6`）。
- **マップ全体配分**：商人 3 個固定、エリート 2–4 個、? 6–10 個。
- **1 ルート上の配分**：敵 4–6、エリート 0–2、休憩 1–3、商人 1–2、宝箱 1、? 3–5。
- **連続禁止**：Rest→Rest は不可。
- 制約違反時は最初から再生成、`MaxRegenerationAttempts = 100` を超えれば `MapGenerationException`。

**Act 差分:** Phase 3 では Act 1 相当の config 1 本のみ同梱。Act 2／3 は後続フェーズで差分 JSON を追加。Core は Act 概念を持たず、呼び出し側が config を切り替える。

**Done 判定:**
- 決定性テスト（同一 seed + config で同一マップ）が通る。
- 全エッジが隣接規則（例外込み）を満たすテストが通る。
- FixedRow / RowKindExclusion / Row 1 敵固定 の各テストが通る。
- マップ全体分布・1 ルート制約・連続禁止 の各テストが通る。
- 実現不可能な config で `MapGenerationException` が投げられるテストが通る。
- Server の DI コンテナから `IDungeonMapGenerator` が解決できる（Smoke）。
- 完了タグ: `phase3-complete`。

## Phase 4 — マップ進行（マス選択）とセーブ連動

**目的:** プレイヤーが次マスを選択し、選択確定＝セーブをトリガする動きを作る。

**成果物:**
- `src/Core/Run/RunActions.cs`（`SelectNextNode(RunState, nodeId) → RunState`）
- `src/Core/Run/TileResolution.cs`（`ResolveTile(RunState, map, rng) → RunState + TileEvent`）
  - 本フェーズでは `Unknown` マスを「選択可能な抽選結果」に変換するロジックのみ実装し、戦闘／商人などの具体処理は後続フェーズ。
- `src/Server/Hubs/RunHub.cs`（`MoveToNode(nodeId)` → サーバ側で Core 呼び出し → 保存 → 新状態を push）
- Client: `src/Client/src/screens/MapScreen.tsx`（マップ描画、現在地、選択可能ノードをハイライト）

**ダンジョン内メニューも本フェーズで実装:**
- `src/Client/src/screens/InGameMenuScreen.tsx`（`あきらめる` / `メニューに戻る` / `音量設定`）。
- `あきらめる` → RunState を `RunProgress.Abandoned` にして保存、プレイ履歴へ送ってメイン画面に戻る。
- `メニューに戻る` → 現在のセーブをそのまま残してメイン画面に戻る（次回 `続きから` で復帰）。
- `音量設定` → Phase 2 の `SettingsScreen` を同じ画面コンポーネントで再利用する。

**Done 判定:**
- ノード選択 → セーブファイルが更新されることを Server.Tests で確認。
- 「メニューに戻る → 再ログイン」で最後に選んだマスから再開できることを統合テスト＋手動確認。
- 「あきらめる」を選ぶと次回ログイン時に新規ラン開始になることを手動確認。

## Phase 10.5 — カードテキスト自動文章化（カードカタログ DX）

**目的:** カード `description` / `upgradedDescription` を effects spec から自動生成し、JSON への手書きを廃止する。手書きとの併存期間を経て段階的に置き換える。

**動機:**
- effects と description の二重管理は陳腐化バグの温床（数値変更時に description 修正漏れ）。
- 同じ effect プリミティブから自動文章化できれば、ローカライズも 1 箇所のテンプレート差し替えで対応可能。
- レリック・ポーションの description にも将来同じ仕組みが流用できる（共通 formatter）。

**スコープ案 (TDD で詰める前のスケッチ):**
1. `Core/Cards/CardTextFormatter` (純関数) を新設。`CardDefinition` + 状態フラグ (upgraded) を入力に、表示文字列を返す。
2. effect プリミティブ単位の文言テンプレートを一覧化（attack / block / draw / discard / debuff / buff / heal / summon / cost-related / 条件発動）。Single/Random/All スコープ、Strength 補正の表記、複数 effect の連結ルール（「。」/ 改行）を仕様化。
3. JSON の description は「override」として残す（自動生成では表現できない特殊カードの逃げ道）。CardJsonLoader が override > 自動生成 の優先順で解決。
4. Server DTO は文字列 description をそのまま流す（Core 側で確定）。Client は受け取った文字列を表示するだけ。
5. Bestiary / 図鑑 / 報酬 / 休憩強化プレビュー など、現状 description を読んでいる全ての UI が自動的に新文言に差し変わることを確認。
6. 既存テスト: card 単位のスナップショットテストで、effects → description が安定して同じ文字列になることを保証。

**着手タイミング:** Phase 10.3-MVP の視覚 UX 修正がひと段落し、カード追加が増えてきて手書き description の保守コストが顕在化したタイミング。Phase 9（マルチ）と独立に進められる。

**設計書 (2026-05-01 確定)**: `docs/superpowers/specs/2026-05-01-phase10-5-design.md`

## Phase 10.6 — Relic / Card Trigger / Modifier 拡張（後日）

**目的:** 現在 9 値の `RelicTrigger` 列挙体に対して、より細粒度なイベントトリガと、計算式に介入する継続補正 (modifier) を追加する。これにより、Slay the Spire ライクな複雑なレリック・パッシブ効果が表現可能になる。

**着手タイミング:** Phase 10.5 (relic / potion / enemy / unit の dev tool 整備) が完了し、editor で全種別を編集できる状態になってから。

**Sub-phase 分割:**

| sub | 内容 |
|---|---|
| **10.6.A** | トリガー型イベント追加 (BeforeHpLoss / AfterHpLoss / OnCardDiscarded / OnCardExhausted / OnEnterShop / OnEnterRestSite / OnRest / OnRewardGenerated / OnCardAddedToDeck / BeforeAttackDamageCalc / BeforeGainBlock / BeforeApplyStatus 等、約 13 個)。enum 拡張 + engine の各 fire point 増設のみで実装可能。各イベントの fire point は 1 箇所程度なので比較的容易 |
| **10.6.B** | 常時補正型 (modifier) アーキテクチャ拡張。effect の新 action として `modifyDamageDealt` / `modifyDamageTaken` / `modifyCardCost` / `modifyShopPrice` / `modifyGoldGain` 等 (約 15 種) を追加。engine の各計算 site (ダメージ計算 / ブロック計算 / カードコスト読取 / status 付与量 / ショップ価格 / etc) で `OwnedRelicIds` を走査して該当 modifier action を集約する仕組みを導入 |

**A: 容易・追加のみ**: 既存 architecture をほぼ変更せず、enum + fire point 増設で完結。
**B: アーキテクチャ拡張**: 計算 pipeline 全域に介入する仕組みなので影響範囲広い。Phase 10.5 完了後の手動プレイテストで挙動を見ながら段階的に進める。

**本ロードマップの段階ではスコープ外。** 着手時に詳細計画 `docs/superpowers/plans/<date>-phase10-6A-relic-triggers.md` 等を起こす。
