# Roguelike Card Game 実装ロードマップ

> **このドキュメントはロードマップ（上位計画）です。** 各フェーズの実装は本ファイルとは別に `docs/superpowers/plans/2026-04-20-phaseNN-<name>.md` として TDD 粒度の詳細計画を書き起こしてから着手します。詳細計画が存在するフェーズのみ着手可能。

**Goal:** Slay the Spire 風のソロ／協力両対応ローグライク（PvE）を、まずは「バトル以外」で一周できる状態まで仕上げ、その後に本格バトルを差し込む。

**Architecture:**
- `Core`（C# .NET 10, 純粋ロジック, Udon# 移植対象）に全ゲームルール／状態遷移／乱数／セーブデータ構造を集約。
- `Server`（ASP.NET Core 10 + SignalR）は Core を呼ぶ薄いアダプタ。シングルプレイでも同一エンドポイント経由。
- `Client`（React 19 + TypeScript + Vite）は表示とユーザー入力のみ。状態は Server から取得。
- マスター定義（カード／レリック／ポーション／モンスター）は `src/Core/Data/*.json` にまとめ、Core 起動時に読み込む。

**Tech Stack:** C# .NET 10, ASP.NET Core 10, SignalR, xUnit, React 19, TypeScript, Vite

**前提ルール（全フェーズ共通）:**
- Core は ASP.NET Core / ネットワーク / ファイル I/O への依存を持たない。JSON 読み込みは「文字列 → モデル」のパース関数として実装し、I/O は呼び出し側で行う。
- record ベースの不変データ。状態遷移は「現在の状態 ＋ 入力 → 新しい状態」の純関数として実装する。
- 乱数は `IRng`（インターフェース）を注入する。テストでは決定的 RNG を使用する。
- テストファースト。まず失敗テスト → 実装 → テスト通過 → コミット。

## 仕様サマリ（ご提示内容を Core モデルに落とし込む前提で整理）

### ゲームループ
1. 起動 → ログイン画面
2. メイン画面：`シングルプレイ` / `マルチプレイ` / `設定` / `実績` / `終了`
3. シングルプレイ選択 → 前回セーブがあれば再開、無ければ新規ラン開始
4. ダンジョン = 3 層。各層はスタート → 分岐路（各ルート 15 マス） → ボス。
5. 各マスで種別に応じたイベント。カード／レリック／ポーション／ゴールドでデッキを強化。
6. 3 層のボスを倒すとクリア。HP 0 でゲームオーバー。どちらも最終結果を保存。

### マス種別
- `?` ランダム（選択式イベント or 商人／宝箱／休憩／敵マスと同等）
- `商人` カード／レリック／ポーションをランダム販売、有償で 1 枚カード廃棄（各マス 1 回）
- `宝箱` ゴールド＋レリック
- `休憩` HP 回復 or カード 1 枚強化
- `敵` 層の通常敵（同層最初の 3 戦は弱プール、4 戦目以降は強プール）
- `エリート敵` 層のエリートプール
- `ボス` 各層終点固定

### 戦闘（暫定・自動勝利）
- 戦闘開始時に現在デッキを「ディープコピー」してバトル用デッキを作り、バトル用デッキに対して戦闘中の加除を行う。
- バトル終了時にバトル用デッキは破棄し、ラン側デッキには影響を与えない。
- バトル以外（休憩の強化／商人の廃棄／イベントでの消去）はラン側デッキを直接変更する。
- **本フェーズではバトル処理を行わず即勝利扱い**とし、通常戦闘／エリート／ボス全てで同じ報酬フローを実行する。

### 報酬
- 敵撃破：ゴールド、ポーション（出現率に従い 0〜1 個）、カード報酬（3 択から 1 枚まで、スキップ可）。
- 宝箱：ゴールド＋レリック 1 個。

### ポーション
- 使い切り。戦闘中・非戦闘中を問わず、発動条件を満たせば使用可。使用で消費。

### レリック
- 入手時効果のみのものと、所持中常時効果のものが混在。消去・変化しない限り永続。

### セーブ
- 自動セーブ。トリガは「マス選択 → そのマスのイベント確定時点」。
- クリア／ゲームオーバー時も保存。
- ダンジョン開始後のプレイ時間を秒単位で保存。再開時は続きから加算。
- ログイン単位で 1 つのセーブを保持（現状ローカル、将来クラウド化）。

### 初期デッキ
- 固定 10 枚からスタート（内容は後で定義）。

### メニュー
- メイン画面 → 設定：マスター音量 / BGM / SE / 環境音。
- メイン画面 → 実績：カード／レリック／ポーション／モンスター図鑑、プレイ履歴。
- ダンジョン内メニュー：あきらめる（ラン放棄）、メニューに戻る（最終保存状態から再開可）、音量設定。

### マルチプレイ
- ホスト：合言葉形式でロビー作成。
- ゲスト：合言葉入力で既存ロビー参加。
- 具体的な同期ルールは本格バトル実装時に合わせて詰める（Phase 10）。

## Phase 0 — JSON データ定義と Core データモデル

**目的:** カード／レリック／ポーション／モンスター／マスタ JSON スキーマを決め、Core にロードする仕組みを作る。

**成果物:**
- `src/Core/Data/Cards/` 配下にカード JSON（初期デッキ 10 枚分＋基本プール最小限）
- `src/Core/Data/Relics/` 配下にレリック JSON（入手時効果型と永続効果型を各 2 件以上）
- `src/Core/Data/Potions/` 配下にポーション JSON（数点）
- `src/Core/Data/Enemies/` 配下に敵 JSON（1 層の弱／強プール＋エリート＋ボス）
- `src/Core/Cards/CardDefinition.cs`（record）
- `src/Core/Relics/RelicDefinition.cs`（record）
- `src/Core/Potions/PotionDefinition.cs`（record）
- `src/Core/Enemy/EnemyDefinition.cs`（record）
- `src/Core/Data/DataCatalog.cs`（文字列群を受け取り各 Definition 辞書を生成する純粋関数）
- `tests/Core.Tests/Data/DataCatalogTests.cs`

**ゲームデザイン決定事項（このフェーズで固める）:**

**カード定義（ユーザー決定）:**
- `id`: 一意の英数字 ID（内部の呼び出し・検索はすべてこれを使う）。
- `name`: カード名。
- `displayName`: 表示名。任意。通常カードでは null、プロモや同一カードのスキン違いで差し替える。未設定なら `name` を表示。
- `rarity`: `0=プロモ / 1=コモン / 2=レア / 3=エピック / 4=レジェンダリー`。
- `cardType`: `Unit / Attack / Skill / Power`。
- `cost`: 通常 0〜3、4 以上も許容。`null` 可（`null` の場合はプレイ不可＝条件付きでしかプレイされないカードを表現）。
- `effects`: 効果プリミティブの配列（例: `[{"type":"damage","amount":6}]`）。本フェーズでは実行はしないが、JSON スキーマと C# モデルだけ先に決める。
- `upgradedEffects`: 強化時の効果配列。未設定なら強化不可。

**初期デッキ（ユーザー決定）— 計 10 枚:**
- `Strike` ×5：`cardType=Attack` / `cost=1` / `effects=[damage:6]` / `upgradedEffects=[damage:9]`
- `Defend` ×5：`cardType=Skill` / `cost=1` / `effects=[gainBlock:5]` / `upgradedEffects=[gainBlock:8]`

**レリック:**
- `trigger`（`onPickup` / `passive` / `onBattleStart` / `onMapTileResolved` 等）と `effect` の組で表現。

**モンスター:**
- HP レンジ、行動テーブル（後で使う）、出現プール（`act1_weak` / `act1_strong` / `act1_elite` / `act1_boss` 等）をタグで保持。

**Done 判定:**
- `DataCatalog.LoadFromStrings(...)` に JSON 文字列を渡して全種の辞書が得られるテストが通る。
- 未知のフィールドを無視し、壊れた JSON では明示的な例外／`Result` を返すテストが通る。

## Phase 3 — ダンジョンマップ生成

**目的:** 「スタート → 分岐路 15 マス → ボス」を 3 層分生成する決定的アルゴリズムを Core に実装。

**成果物:**
- `src/Core/Map/MapNode.cs`（座標、マス種別、接続先ノード ID 配列）
- `src/Core/Map/TileKind.cs`（`Enemy` / `EliteEnemy` / `Merchant` / `Treasure` / `Rest` / `Unknown` / `Start` / `Boss`）
- `src/Core/Map/DungeonMapGenerator.cs`（`Generate(seed, actNumber)` → `DungeonMap`）
- `src/Core/Map/DungeonMap.cs`
- `tests/Core.Tests/Map/DungeonMapGeneratorTests.cs`

**生成規則（このフェーズで確定）:**
- 1 層あたり「スタート 1 マス＋中間 15 マス＋ボス 1 マス」。
- 分岐は数本のパスがあり、どのパスもスタート→ボスに到達可能。
- 同じ seed・同じ act 番号からは常に同一のマップが生成される（テストで保証）。

**固定制約（ユーザー決定）:**
- **9 マス目 = 宝箱（各層唯一）**：どのルートを通っても 9 マス目に到達すると必ず宝箱マスになる。層内の他の位置に宝箱は出さない。実装方針は「9 マス目は分岐を全て 1 ノードに合流させる」STS 型レイアウト。
- **15 マス目 = 休憩（ボス前）**：どのルートを通ってもボス 1 つ前のマスは休憩で固定。9 マス目と同じく 15 マス目も 1 ノードに合流させる。
- **商人 = 1 層あたり 3 マス**：層内の商人マスの総数はちょうど 3。

**分布（9／15／商人を除いた残りの分布 — 暫定）:**
残り 11 マス × 複数ブランチぶん。敵 58％ / ？ 22％ / エリート 10％ / 休憩 10％。ここは Phase 3 でバランス調整。

**局所制約:**
- スタート隣接マスは `敵` 固定。
- 連続 `休憩` 禁止、連続 `エリート` 禁止、連続 `商人` 禁止。
- エリートはマス 1〜5 には出さない（序盤強敵回避）。

**Done 判定:**
- 同一 seed で複数回生成して一致するテストが通る。
- 任意のノードからボスへ到達可能であることを検証するテストが通る。
- 15 マス制約・局所制約が守られていることを検証するテストが通る。

## Phase 9 — マルチプレイ・ロビー（戦闘無しでの進行同期）

**目的:** 合言葉形式でロビーを作成／参加し、**マップ進行のみ**を同期する最小実装。戦闘同期は Phase 10 で作る。

**前提（ユーザー決定）:**
- マップはパーティ共通（全員同じマップを見る）。
- 次マスの決定は「全プレイヤーが自分の選びたいマスを選んだら、選択されたマスの中から 1 つを一様ランダムで選出」。
- イベント内容・敵の抽選も完全にパーティ共有。全員が同じ敵と協力して戦う（同じマスに入れば同じ報酬抽選卓に乗る）。

**成果物:**
